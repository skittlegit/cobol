## Objective

Implement `verify(finding, tools) -> VerificationResult`: attempt evidence in
tier order, record the tier that succeeded, and **reject unsupported
citations**. This is the gate that separates a finding from a fluent guess, and
CONTRACT.md Part 3 requires T4 faithfulness to be reported *per tier* — so the
tier recorded here is load-bearing for the headline metric, not decoration.

## Models

```python
class VerificationTier(IntEnum):
    EXECUTED = 1      # behaviorally executed via run_cobol
    STATIC = 2        # AST/dataflow/slice fact matches the claim
    ENTAILMENT = 3    # retrieved clause entails the claim (NLI)

class VerificationResult(BaseModel):
    model_config = ConfigDict(extra="forbid")
    verified: bool
    tier: VerificationTier | None       # None iff verified is False
    evidence: str                       # what was checked, quotable in a trace
    citation_ok: bool                   # did the cited clause survive checking
    rejected_reason: str | None = None
    tier_attempts: list[TierAttempt]    # every tier tried, in order, w/ outcome
```

`tier_attempts` is required, not optional: T4.4 calibration and the T4.3
trajectory eval both need to know a Tier-1 attempt was *tried and unavailable*
versus never attempted. Silent tier downgrade is exactly the failure the
playbook's risk register calls out for CICS-heavy code.

## Tier semantics

**Tier 1 — executed.** Applicable when the finding's locus is compilable and
runnable. Use `compile_check` first; on success, `run_cobol` with inputs that
exercise the claimed boundary (for D1/D5, values on both sides of the
threshold). Verified iff observed behavior matches the claim. **Most
CardDemo/CBSA online code is CICS-heavy and will not compile** — that is an
expected `unavailable` attempt, not a failure. Record it as such.

**Tier 2 — statically confirmed.** The claim is matched against a fact from the
tool layer: the literal (`read_paragraph` / `slice_on`), the comparator
(`slice_on`), the def-use chain (`trace_variable`), or reachability. Verified
iff the tool-reported fact matches the finding's claim.

**D6 reachability — mandatory, from Track A's 2026-07-17 flag.** D6 dead-code
verification must consume **`forest_roots` plus `reachable_from`**, and must
**not** use `entry_points`, whose meaning changed to the single true program
entry. Reachability traverses `edge_kind="fallthrough"`. Getting this wrong
silently produces wrong D6 verdicts at M4. Assert it in a test.

**Tier 3 — entailment only.** NLI check that the retrieved clause entails the
claim. **Different model family than the system under test** (CONTRACT.md
integrity rule). If the system under test is Claude, the verifier must not be
Claude. Pin model + provider as module constants and record them in the
accuracy report. Tier 3 is the weakest tier and must be flagged as such on the
result.

**Ordering rule:** attempt 1 → 2 → 3, stop at first success. Record every
attempt. If all three fail, `verified=False`, `tier=None`, and the agent
abstains (T3.5 consumes this).

## Citation rejection

Independent of tiering: if the finding cites a clause that does not support the
claim, `citation_ok=False` and the finding is rejected **even when a tier would
otherwise pass**. A correct code fact attached to the wrong clause is exactly
the "plausible but wrong" failure the entailment check exists to stop.

## Gates

1. Each fixture verifies at its expected tier; `tier_attempts` records the full
   ladder including `unavailable` Tier-1 attempts.
2. The planted unsupported citation is **rejected** (`citation_ok=False`).
3. Tier ordering: a finding that could verify at Tier 2 does **not** silently
   report Tier 3; a Tier-1-unavailable finding falls through to 2, not to 3.
4. D6 uses `forest_roots` + `reachable_from`; a test asserts `entry_points` is
   **not** consulted.
5. Family separation: a test asserts the Tier-3 verifier's configured model
   family differs from the system-under-test constant.
6. Offline determinism: Tier 3 responses cached to
   `tests/fixtures/verify/nli_cache.jsonl` (committed) so the suite runs
   offline and reproducibly, same pattern as the T3.3b HyDE cache.

## Verifier accuracy report and evidence (required — not optional)

The playbook and CONTRACT both require the Tier-3 verifier to be
**human-validated on a sample, with its accuracy reported**, because NLI on
long deontic legal text is unreliable and an unvalidated verifier is itself an
unbacked claim. This section is the report of record.

- ≥50 (claim, clause) pairs drawn from `clauses.jsonl` + seed instances, with a
  deliberate mix of entailed / not-entailed / partially-relevant.
- Verifier verdict vs human label; report accuracy, and separately the
  **false-accept rate** (unsupported claims wrongly accepted) — that is the
  number that matters, since false accepts become hallucinated citations.
- **Claude Code generates the pairs and runs the verifier; it does NOT author
  the human labels.** Commit the pair set with a `human_label: null` column;
  the labels are filled at review in the Track C chat. Mark the accuracy tests

### Metrics after human labeling

- **Accuracy:** `(verdict == human_label) / 50`.
- **False-accept rate:** `(verdict entails and human_label is not-entail) /
  (human_label is not-entail)`.

The false-accept rate is the load-bearing metric because an unsupported claim
accepted by the verifier becomes a hallucinated citation. Once labels exist
and the neural cache is regenerated,
`test_verifier_accuracy_false_accept_rate` becomes a hard gate with a maximum
false-accept rate of 10%.
