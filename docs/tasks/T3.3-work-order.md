## Precondition

T3.2 must replace its historical 1,824-chunk Gate B evidence with a passing run
on the current 2,361-chunk/8-document fixture. Do not tune HyDE against a stale
retrieval baseline.

## Objective

Retrieve on a generated natural-language description of the code slice ("what
rule does this implement?") instead of the raw query (variable names / raw COBOL
tokens). Reuse the T3.2 engine: `RegulationIndex.search(query, mode=...)` over the
same fixed eval set (`tests/fixtures/retrieval/queries.jsonl`). HyDE becomes a
query-transform stage in front of the existing modes.

**Done when:** measurable retrieval gain vs the raw query on the
same eval set — report the T3.2 metric table (hit@1, hit@3, mrr@5 across
bm25/dense/hybrid/hybrid_rerank) for raw-query vs HyDE-query, and show HyDE
strictly beats raw on mrr@5 for at least the dense and hybrid_rerank modes.

## Notable findings carried from T3.2 (the reason this task is well-motivated)

Grounded in the T3.2 work order's historical relevance evidence (24 queries,
1824-chunk corpus,
pinned bge-small + MiniLM cross-encoder):

**For T3.3 (HyDE):**

1. **The dense channel is the weak link, and it fails exactly where HyDE helps.**
   Dense (bge-small) trails BM25 on this set — mrr@5 **0.592 vs 0.635**. The
   clearest case is the raw-code confusion probe **q20** (`IF WS-ACTV-FLAG = 'N'
   AND WS-DAYS-SINCE-ISSUE > 30 ...`, gold CC-06a-vi): dense **missed entirely
   (rank –)** while bm25=2 and hybrid_rerank=1. Raw COBOL identifier/token forms
   are out-of-distribution for a sentence embedder — precisely the input HyDE
   replaces with an NL description. Prioritise the raw-code / identifier-heavy
   queries when measuring HyDE gain.
2. **q23 (KYC-ckycr-update) misses in all four modes** — the "7-day CKYCR update"
   vs "7-working-day account-closure" clause collision defeats lexical and dense
   retrieval alike. A HyDE description that names the *entity + obligation*
   ("update central KYC registry after receiving customer info") is the intended
   escape hatch; treat q23 as the canary for whether HyDE resolves near-duplicate
   deadline clauses.

**For T5.5:**

3. **The cross-encoder reranker is non-monotonic per query.** It wins on aggregate
   (mrr@5 0.744, best mode) but *regresses* specific queries vs plain hybrid:
   **q14 1→5, q17 1→2, q19 1→2, q24 1→(–)**. Any downstream stage that consumes
   ranked retrieval must not assume the reranker only helps; report per-query
   deltas, not just the mean. (**Note:** T5.5 is not yet defined in any track
   work order — this finding is parked here for whoever scopes it; re-file if T5.5
   lands elsewhere.)
