## Part 1 — `search_regulations` wiring

### The chunk→clause mapping (the real work — not a pass-through)

`RegulationIndex` returns **`RegulationChunk`s**; `RegSearchHit` wraps a
**`RegulationClause`**. So the service must map each retrieved chunk back to its
source clause record:

- Join key: chunk `(doc, clause_id)` → the record in `clauses.jsonl` whose
  `clause.doc`/`clause.clause_id` match. Build the lookup once at service init
  from `data/regulations/clauses.jsonl`; the resulting `RegulationClause` is the
  record's `clause` sub-object (carries `current_value`, which chunks lack and
  the agent needs).
- **`clause_id is None` chunks** (front matter, annexes, definitions with no
  clause number) **cannot** map to a clause record. Drop them from results —
  the tool contractually returns clause-anchored hits. This is expected
  behavior, not a gap; assert it in a test so it isn't "fixed" later.
- **Deduplication:** multiple chunks can map to the same clause_id (a
  multi-chunk clause). Collapse to one `RegSearchHit` per clause, keeping the
  best (max) score. `search_regulations` returns clauses, not chunks.
- Score: pass through the retrieval score of the best chunk for that clause.
  (F8 is fixed, so hybrid scores are real, not 0.0.)

### `RegulationSearch` API

```python
class RegulationSearch:
    def __init__(self, chunks_path, clauses_path,
                 mode="hybrid_rerank", use_hyde=False): ...
    def search(self, query: str, k: int = 5) -> list[RegSearchHit]: ...
```

`RealToolLayer.search_regulations(query)` constructs (or holds) a
`RegulationSearch` with the pinned default (`hybrid_rerank`, `use_hyde=True` —
see Part 2 decision) and returns `search(query, k=5)`.

### Gates (offline where possible)

- **Mapping:** every returned `RegSearchHit.clause` is a real record from
  `clauses.jsonl`; no `clause_id=None` leaks; dedup holds (no clause appears
  twice).
- **`isinstance` parity:** `RealToolLayer` still satisfies `ToolLayer`
  (`search_regulations` no longer raises).
- **Contract shape:** returns `list[RegSearchHit]`, len ≤ k.

## Part 2 — HyDE query generation

### What HyDE does here

The agent's raw query is often a COBOL slice or a bag of identifiers
(`WS-YEARS-SINCE-KYC`, `WS-RISK-RATING`). HyDE first has an LLM draft a short
*hypothetical regulatory clause* that such code would implement, then retrieves
on **that** text instead of the raw identifiers — closing the vocabulary gap
between code and deontic legal prose.

`hyde.py`:
- `generate_hypothesis(query: str) -> str` — one LLM call, tightly prompted to
  emit 2–3 sentences of plausible regulation prose, no preamble. Pin the model
  + a low temperature; record both as module constants.
- Feeds the hypothesis to `RegulationIndex.search`. Optionally concatenates
  hypothesis + raw query (ablation flag `hyde_concat: bool`).
- **Determinism:** cache `generate_hypothesis` by query hash to a committed
  `tests/fixtures/hyde/hypotheses.jsonl` so the eval is reproducible offline
  and the network gate isn't required at review. Generate live once, commit the
  cache, then tests read the cache.

### Evaluation — the bar

Re-run the 24-query relevance eval with `use_hyde=True` vs the T3.2 baseline
(`use_hyde=False`), same `hybrid_rerank` mode, same gold resolution. Emit a
mode-comparison table (baseline vs HyDE vs HyDE-concat) with hit@1/hit@3/mrr@5,
the per-query appendix, and the q20–q24 sub-table.

**Named target:** report q23 (KYC-ckycr-update) explicitly — it missed in all
four raw modes across both the 7- and 8-doc corpora. HyDE lifting q23 into the
top-5 is the specific, defensible improvement claim; if it doesn't, say so.

**Done-when bar (honest, not inflated):** HyDE **does not regress** aggregate
mrr@5 vs the T3.2 baseline, **and** the q23 outcome (lifted or still-missed) is
reported either way. HyDE is a targeted intervention — a small aggregate move
with q23 recovered is a success; a large regression is a fail even if q23 is
recovered. If HyDE regresses aggregate mrr@5, default `use_hyde=False` in the
tool, commit the evidence, and flag for review — **do not** ship a retrieval
default that's worse than T3.2.
