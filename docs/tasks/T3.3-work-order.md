## Part A — T3.3a: `search_regulations` wiring

**Status:** done — reviewed 2026-07-24 and merged in PR #61.

### Objective

Expose clause-anchored regulation search through the frozen `ToolLayer`
contract. The T3.2 index returns ranked `RegulationChunk` objects, while
`search_regulations` must return `RegSearchHit` objects containing complete
`RegulationClause` records. Part A owns that mapping and the real tool-layer
delegation; it does not implement HyDE.

### Delivered files

- `src/cobol_archaeologist/rag/search.py` — `RegulationSearch`, clause lookup,
  and chunk-to-clause hit mapping.
- `src/cobol_archaeologist/tools.py` — lazy
  `RealToolLayer.search_regulations` delegation.
- `tests/test_search_regulations.py` — offline mapping, contract-shape, and
  delegation gates.

### Part A gates and durable evidence

1. Every returned hit is anchored to a real `(doc, clause_id)` record from
   `data/regulations/clauses.jsonl`.
2. Unnumbered chunks (`clause_id=None`) and chunks without an anchored clause
   record are dropped.
3. Multiple chunks from one clause collapse to one result, preserving
   first-appearance rank and the maximum score.
4. Results are `list[RegSearchHit]` with `len(results) <= k`.
5. `RealToolLayer` remains structurally compatible with `ToolLayer`, and its
   method delegates to the search service instead of raising
   `NotImplementedError`.
6. The forward-compatible `use_hyde` option is accepted but remains inert until
   Part B, so Part A cannot silently claim HyDE behavior.

## Part B — T3.3b: HyDE query generation

**Status:** done — implementation and pinned-model gates reviewed; merged in
PR #67.

### Objective

Retrieve on a generated natural-language description of the code slice
("what rule does this implement?") instead of the raw query (variable names and
raw COBOL tokens). HyDE is a query-transform stage in front of the existing T3.2
search modes; it does not replace the index or Part A's clause mapping.

### Completion bar

Report raw-query versus HyDE-query hit@1, hit@3, and mrr@5 for
`bm25`, `dense`, `hybrid`, and `hybrid_rerank` on the same fixed evaluation set.
HyDE must:

1. produce a measurable retrieval gain over raw queries;
2. strictly beat raw-query mrr@5 in at least the `dense` and
   `hybrid_rerank` modes;
3. report per-query deltas, with raw-code/identifier-heavy queries called out;
4. treat q23 (`KYC-ckycr-update`) as the named canary for resolving the
   near-duplicate 7-day deadline clauses; and
5. keep every Part A gate green.

### Notable findings carried from T3.2

The historical diagnostic below used the 24-query, 1,824-chunk corpus with the
pinned bge-small embedder and MiniLM cross-encoder. The completion metrics must
come from the current 2,361-chunk/8-document fixture.

1. **The dense channel is the weak link, and it fails where HyDE should help.**
   Dense historically trailed BM25 on this set (mrr@5 0.592 vs 0.635). For the
   raw-code confusion probe q20 (`IF WS-ACTV-FLAG = 'N' AND
   WS-DAYS-SINCE-ISSUE > 30 ...`, gold CC-06a-vi), dense missed while BM25
   ranked it second and hybrid-rerank ranked it first. A natural-language
   description should remove the raw COBOL identifier mismatch.
2. **q23 missed in all four modes.** The "7-day CKYCR update" and
   "7-working-day account closure" clauses collide lexically and semantically.
   The HyDE description should name the entity and obligation: updating the
   central KYC registry after receiving customer information.
3. **The reranker is non-monotonic per query.** It historically regressed q14
   1→5, q17 1→2, q19 1→2, and q24 1→miss versus plain hybrid despite winning on
   aggregate. Downstream work must use per-query deltas rather than assuming
   reranking always helps. This finding is also relevant to the future T5.5
   scope.
