# T2.4 Plausibility-Judging Rubric

## Purpose

Judge whether a mutated COBOL locus resembles regulatory drift that a real
legacy maintainer could leave behind. This is not a correctness re-annotation:
the clause and mutation label are already fixed. The question is whether the
code shape is credible enough for benchmark use.

## Verdicts

- **plausible** — A realistic maintenance history could produce the shown
  code. The change fits surrounding COBOL style, uses believable values or
  control flow, and leaves no conspicuous mutation-generator fingerprint.
- **implausible** — The code looks mechanically damaged or contrived: absurd
  values, malformed/pointless structure, sharp style discontinuity, unrelated
  edits at the labeled locus, or a mutation no maintainer would reasonably
  leave behind.
- **unsure** — Credibility depends on missing operational context or domain
  facts. Do not turn uncertainty into a rejection; send it to human review.

## Review dimensions

1. **Legacy-maintenance story:** Can a stale constant, missed branch, inverted
   gate, old reference list, boundary choice, or disabled flag arise through a
   normal incomplete change?
2. **Style continuity:** Do names, indentation, fixed-format layout, comments,
   paragraph structure, and literal formatting resemble the surrounding file?
3. **Magnitude and semantics:** Are substituted values near a real historical
   or operational value rather than absurd? Does the edit affect the regulated
   behavior at the displayed locus?
4. **Artificiality signals:** Reject unmatched syntax, wholesale unrelated
   deletion, sentinel text announcing a mutation, or edits whose only purpose
   is to satisfy a benchmark label.

The judge sees only clause text and mutated locus context. Gold rationale,
drift label, mutation operator, and provenance are withheld to prevent leakage.
The set also contains conformant controls: “plausible” means the displayed code
or maintenance edit looks natural, not that every excerpt must violate the
clause. A natural conformant control is plausible; an obviously generator-made
control is implausible.

## Worked seed-corpus examples

### Example 1 — BOIDENT1 stale beneficial-owner threshold

`BOIDENT1` retains the former 15-percent partnership threshold after the KYC
rule moved to more than 10 percent. Verdict: **plausible**. A historical value
surviving a regulatory amendment is a common legacy failure, the magnitude is
documented rather than arbitrary, and the surrounding comparison remains
ordinary COBOL.

### Example 2 — LATEFEE1 charges the total amount due

`LATEFEE1` computes the late charge from `WS-TOTAL-AMT-DUE` instead of the
outstanding amount after the due date. Verdict: **plausible**. Both fields are
nearby business amounts with compatible shapes, so choosing the older/wrong
base resembles a genuine incomplete rule migration rather than a synthetic
type error.

### Example 3 — CLOSPEN5 penalty path disabled by a flag

`CLOSPEN5` contains the penalty paragraph but initializes its enabling flag to
`N`, leaving the compliance calculation dead. Verdict: **plausible**. Pilot and
feature flags commonly survive rollout, the disabled value matches the data
item's representation, and the guarded paragraph remains stylistically
consistent. An unexplained random flag name or impossible value would instead
be **implausible**.

## Human spot-check protocol

Review exactly 15 unique instances from the stratified 50-item sample. Record
`instance_id`, one of the three verdicts, and a short reason in JSONL. Agreement
is exact verdict agreement with the external judge; disagreements remain
visible and every `unsure` case must be adjudicated before T2.4 can close.

## Run sequence

Configure `OPENAI_API_KEY` (or select another key variable with
`--api-key-env`), `--model`, `--model-family openai|google`, and an
OpenAI-compatible `--endpoint` when needed. For reasoning models, select an
explicit `--reasoning-effort`; the client omits temperature because reasoning
requests reject non-default sampling values. Then run:

```text
cobol-archaeologist benchmark-judge --sample 50 --model <model> --model-family <family> --reasoning-effort <effort>
cobol-archaeologist benchmark-judge --model <model> --model-family <family> --reasoning-effort <effort>
cobol-archaeologist benchmark-judge --adjudications <all-unsure-decisions.jsonl>
cobol-archaeologist benchmark-judge --human-review <15-row-human-review.jsonl>
```

The sample command records the gate evidence. The full command writes the
plausible-only output plus `rejected/{implausible,unsure}.jsonl`. The final
command records exact human agreement in the benchmark manifest.

The adjudication file covers every and only `unsure` judgement and records a
human `plausible` or `implausible` verdict plus a reason. The original external
verdict remains unchanged in `judgements.jsonl`; adjudication updates the
plausible output/rejected sidecars and is summarized in the manifest.

Judging checkpoints each completed verdict. Rerunning with the same output
resumes a matching prefix. If a corrected benchmark retains unchanged instance
IDs, `--reuse-judgements <prior.jsonl>` safely reuses only rows whose ID, drift
type, stratum, model, and family still match; changed instances are sent to the
endpoint normally.
