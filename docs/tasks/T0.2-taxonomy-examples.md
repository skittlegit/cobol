Status: **taxonomy v1 LOCKED**. Each class below is grounded in a concrete locus

`app/cpy`, line numbers from the current main branch — pin the commit in

Notation per class: the locus, a ≤15-line excerpt, the clause _type_ it maps to,
and the mutation operator that synthesizes it (playbook Part 1C).

## The anchor-fit decision (the open question this task had to answer)

**Decision: option (a), widened anchor clause set.** The benchmark's anchor
clause set is the **RBI Master Direction — Credit Card and Debit Card – Issuance
and Conduct Directions, 2022** (issued 21 Apr 2022, effective 1 Jul 2022;
amended 7 Mar 2024 — T2.1 must re-verify the "updated as on" state from
rbi.org.in at curation time), _plus_ the KYC/AML clauses that its **clause 20
incorporates by reference** from the RBI KYC Directions. The **KYC Directions
2025** remain the anchor for the real-curated seed (T2.5) and the T6
versioned-judgment pairs; KYC-only logic with no CardDemo host (e.g. periodic
re-KYC scheduling) is implemented in the GnuCOBOL-native runnable base — a
limited slice of option (b), used only where (a) leaves a gap.

Three-sentence justification. CardDemo is a credit-card servicing system, so the
CC Directions' conduct rules (interest/APR methodology, over-limit treatment,
late-payment charges, account-closure timelines, card activation/expiry) map
onto its real code paths, whereas pure-KYC checks mostly do not. The union is
not arbitrary: CC Directions clause 20 makes KYC/AML/CFT compliance a binding
obligation of card issuers, so a KYC-referencing clause set over card-servicing
code is legally coherent, and the 7 Mar 2024 amendment (₹500/working-day →
₹500/calendar-day closure penalty; new "Total Amount Due" definition) supplies a
**second real regulation-change pair for T6** alongside KYC 2016→2025. This
keeps CardDemo as the single benchmark anchor (license, tooling, and runnability
already validated) instead of fragmenting the corpus.

**CONTRACT CHANGE — affects Tracks B/C.** T2.1's clause-set scope is now: CC
Directions 2022 (as amended 2024) primary + KYC 2025 clauses reachable via
clause 20 + KYC 2025 as the T2.5/T6 seed source. Track C impact: T3.1 chunker
corpus is two regulation documents, not one; T0.4 metrics-half can proceed
against this scope. No change to the DriftInstance schema —
`regulation_clause.doc` already carries the document identity.

## D1 — Stale parameter / threshold

**Locus:** `CBACT04C.cbl`, `1300-COMPUTE-INTEREST`, lines 462–468 (rate source:
`1200-GET-INTEREST-RATE` lines 415–440, `DEFAULT` fallback line 437;
`DIS-INT-RATE` declared in copybook `CVTRA02Y.cpy` line 9).

```cobol
       1300-COMPUTE-INTEREST.

           COMPUTE WS-MONTHLY-INT
            = ( TRAN-CAT-BAL * DIS-INT-RATE) / 1200

           ADD WS-MONTHLY-INT  TO WS-TOTAL-INT
           PERFORM 1300-B-WRITE-TX.
```

**Clause type:** interest/finance-charge methodology (CC Directions: APR
quotation and finance-charge calculation methodology must match disclosure). The
`/ 1200` is a hardcoded annualization basis; the disclosure-group `DEFAULT`
fallback rate is a hardcoded regulated value one indirection away. **Operator:**
**MO-1** (replace the regulated literal with an outdated value). Natural
**MO-1×** host: move/keep the rate or basis constant in a copybook (`CVTRA02Y`)
— threshold defined far from its use site, exactly the headline-stratum shape.

## D2 — Missing rule

**Locus:** `CBTRN02C.cbl`, `1500-VALIDATE-TRAN`, lines 370–378. **This is a
naturally-occurring D2 in shipped code**, not just a mutation host: the posting
validator checks card x-ref, credit limit, and expiry — but never checks
`ACCT-ACTIVE-STATUS` (field: `CVACT01Y.cpy` line 6; grep confirms no batch
posting program reads it). The source even ships the comment marking the
absence.

```cobol
       1500-VALIDATE-TRAN.
           PERFORM 1500-A-LOOKUP-XREF.
           IF WS-VALIDATION-FAIL-REASON = 0
              PERFORM 1500-B-LOOKUP-ACCT
           ELSE
              CONTINUE
           END-IF
      * ADD MORE VALIDATIONS HERE
           EXIT.
```

**Clause type:** no transactions on blocked/closed accounts (CC Directions:
closure/blocking obligations; clause-20-referenced KYC: transaction restrictions
on non-compliant accounts). **Operator:** **MO-2** (delete an implementing
branch/PERFORM — e.g. remove the expiry check at lines 414–420 to synthesize
additional D2 instances). The active-status gap is also a candidate
**real_curated** instance.

## D3 — Contradictory / over-permissive

**Locus:** the block/post split in `CBTRN02C.cbl`: the blocker lives in
`1500-B-LOOKUP-ACCT` (lines 407–413), but the _only_ thing standing between a
failed validation and `2000-POST-TRANSACTION` (line 424) is one gate in the main
loop (lines 210–216):

```cobol
                     PERFORM 1500-VALIDATE-TRAN
                     IF WS-VALIDATION-FAIL-REASON = 0
                       PERFORM 2000-POST-TRANSACTION
                     ELSE
                       ADD 1 TO WS-REJECT-COUNT
                       PERFORM 2500-WRITE-REJECT-REC
                     END-IF
```

**Clause type:** over-limit transactions only with cardholder consent /
forbidden postings must not complete (CC Directions over-limit provisions).
**Operator:** **MO-3** (remove/weaken the blocking branch so forbidden cases
pass). Natural **MO-3×**: the contradiction split across paragraphs/programs —
blocker intact in `1500-B`, a second posting path added (or the gate condition
weakened) elsewhere; the validate-then-gate-then-post structure is exactly the
two-locus shape MO-3× needs.

## D4 — Stale reference data

**Locus:** copybook `CSLKPCDY.cpy` — three hardcoded 88-level enumerations:
`VALID-PHONE-AREA-CODE` (line 30 ff.), `VALID-US-STATE-CODE` (lines 1013 ff.),
`VALID-US-STATE-ZIP-CD2-COMBO` (lines 1073 ff.).

```cobol
       01 US-STATE-CODE-TO-EDIT  PIC X(2).
          88 VALID-US-STATE-CODE VALUES
                                    'AL',
                                    'AK',
                                    'AZ',
                                    'AR',
                                    'CA',
```

**Clause type:** validation against a regulator-maintained reference set
(KYC/AML via CC Directions clause 20: sanctioned-jurisdiction /
restricted-entity lists — the enumerated-88-level-in-a-copybook pattern is the
exact code shape such lists take). **Operator:** **MO-4** (remove/alter an
entry). Because the list lives in a copybook consumed by other programs, every
MO-4 here is inherently interprocedural — definition site and use sites are in
different files.

## D5 — Boundary / comparator error

**Locus:** `CBTRN02C.cbl`, `1500-B-LOOKUP-ACCT`, lines 403–413 — the over-limit
boundary; secondary host at line 548 (`IF DALYTRAN-AMT >= 0`, credit/debit cycle
classification at the zero boundary).

```cobol
                COMPUTE WS-TEMP-BAL = ACCT-CURR-CYC-CREDIT
                                    - ACCT-CURR-CYC-DEBIT
                                    + DALYTRAN-AMT

                IF ACCT-CREDIT-LIMIT >= WS-TEMP-BAL
                  CONTINUE
                ELSE
                  MOVE 102 TO WS-VALIDATION-FAIL-REASON
                  MOVE 'OVERLIMIT TRANSACTION'
                    TO WS-VALIDATION-FAIL-REASON-DESC
                END-IF
```

**Clause type:** "exceeds the credit limit" — inclusive vs exclusive treatment
of the exactly-at-limit case (CC Directions over-limit wording). **Operator:**
**MO-5** (`>=` ↔ `>`): flipping line 407 silently changes whether an at-limit
transaction posts — the canonical right-rule-wrong-edge drift.

## D6 — Dead compliance code

**Locus (mutation host):** the expiry check, `CBTRN02C.cbl` lines 414–420 — a
discrete compliance check that MO-6 wraps in an always-false guard. **The guard
idiom to mimic is already in the same program:** `WS-CREATE-TRANCAT-REC`
(declared line 190 `VALUE 'N'`, set line 478, tested line 495):

```cobol
        01 WS-FLAGS.
           05 WS-CREATE-TRANCAT-REC         PIC X(01) VALUE 'N'.
      * ... 300 lines later ...
           IF WS-CREATE-TRANCAT-REC = 'Y'
              PERFORM 2700-A-CREATE-TCATBAL-REC
           END-IF
```

**Clause type:** any coded-but-unreachable required check (e.g. expiry
enforcement present but guarded by a flag no path sets). **Operator:** **MO-6**
(wrap in an always-false guard using exactly this flag idiom). **MO-6×**: the
flag initialized/cleared in a _different_ program that runs earlier in the batch
chain (CBTRN01C → CBTRN02C job sequence) — definition and dead guard in separate
files.

## D7 — Conformant (negative class)

**Locus:** `CBTRN02C.cbl`, lines 414–420 — the expiry check, correctly
implementing "no transactions after account expiration":

```cobol
                IF ACCT-EXPIRAION-DATE >= DALYTRAN-ORIG-TS (1:10)
                  CONTINUE
                ELSE
                  MOVE 103 TO WS-VALIDATION-FAIL-REASON
                  MOVE 'TRANSACTION RECEIVED AFTER ACCT EXPIRATION'
                    TO WS-VALIDATION-FAIL-REASON-DESC
                END-IF
```

**Clause type:** card/account validity enforcement (CC Directions
expiry/validity provisions). Correct rule, correct comparator → the ground-truth
conformant instance. **Operator:** **MO-0** benign edits applied here (renames,
reflow, comment rewording, non-regulated literal changes) so conformant
instances are edited too and "was-this-edited" carries zero label information.
(Bonus realism: the field is literally misspelled `ACCT-EXPIRAION-DATE` in the
shipped copybook — real legacy code noise our benign edits should imitate, not
sanitize.)

## Residual gaps (honest accounting)

1. **Pure-KYC classes have no CardDemo host** (periodic re-KYC scheduling, PEP
   screening as a distinct workflow). Per the fit decision these are hosted in
   the GnuCOBOL-native base and in the T2.5 real-curated seed — they will be
   underrepresented in CardDemo-derived synthetic instances. The DATASHEET
   (T5.2) must state this.
2. **Line numbers drift with upstream CardDemo commits.** T2.1 pins the exact
   commit in `data/manifest.json`; all loci above must be re-checked against the
   pinned commit.
3. **D4's real lists are US-domestic** (state/area codes), standing in for
   regulator-maintained lists. The code _shape_ is exact; the semantic mapping
   to RBI-referenced lists is by analogy. Synthetic D4 instances should
   introduce an RBI-plausible list (e.g. jurisdiction codes) using the same
   88-level copybook idiom.
4. **CC Directions currency:** verified as issued 21 Apr 2022 / effective 1 Jul
   2022 / amended 7 Mar 2024 via rbi.org.in. T2.1 must re-verify no later
   amendment before encoding `version`/`effective_date` into clause records.
