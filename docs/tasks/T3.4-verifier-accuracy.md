# T3.4 — Tier-3 verifier accuracy report

**Status:** pairs generated + verifier run; **human labels PENDING** (Track C
chat). Accuracy tests are `xfail(reason="pending human labels")` until the
`human_label` column is filled, mirroring the T3.1 Gate B protocol.

## Why this report exists

Tier 3 (entailment-only) is the weakest verification tier and the one most
likely to accept a fluent-but-wrong finding: NLI over long deontic legal text is
unreliable, so an **unvalidated** entailment verifier is itself an unbacked
claim. This report is the validation. The number that matters is not headline
accuracy but the **false-accept rate** — unsupported claims the verifier wrongly
accepts — because a false accept becomes a hallucinated citation shipped as
"verified".

## Verifier under test

- **Tier-3 model (of record):** `MoritzLaurer/DeBERTa-v3-base-mnli-fever-anli`
  (`NLI_MODEL`), revision `main`, provider HuggingFace, family **`deberta`**.
- **Family separation (CONTRACT integrity rule):** the system under test is
  Claude (family `anthropic`); the verifier family is `deberta`. `deberta` ≠
  `anthropic` — asserted in `tests/test_verify.py::test_tier3_model_family_differs_from_system_under_test`.
- **Decision rule:** entailment iff `P(entailment) ≥ 0.5` (`ENTAIL_THRESHOLD`).

### Offline proxy caveat (honest provenance)

The pinned DeBERTa weights are **not available in the current build environment**
(no `transformers`/GPU). The committed verdicts below and in
`tests/fixtures/verify/nli_cache.jsonl` are produced by the transparent
`LexicalEntailer` token-coverage proxy (`backend: lexical_proxy_v1`), **tagged as
such** — never presented as neural output. To produce the verdicts of record,
re-run:

```bash
python scripts/build_verify_nli_cache.py --neural
```

which recomputes the cache with the pinned model. The table then reflects the
neural verifier; the human labels are the same either way.

## The pair set

`tests/fixtures/verify/accuracy_pairs.jsonl` — **50 (claim, clause) pairs** drawn
from `data/regulations/clauses.jsonl`, deliberately mixed:

| design intent | count | what it probes |
|---|---|---|
| entailed (`acc_pos_*`) | 21 | leading obligation of the same clause — should be accepted |
| not-entailed (`acc_neg_*`) | 21 | a *different* clause's obligation — should be rejected |
| partially-relevant (`acc_part_*`) | 8 | an adjacent clause in the same instrument — mixed |

`design_intent` records Claude's generation rationale; **`human_label` is `null`**
and is authored only at review. Claude generated the pairs and ran the verifier;
Claude did **not** author the human labels.

## Verifier verdicts (preliminary — `lexical_proxy_v1`)

| design intent | verdict = entail | verdict = not-entail |
|---|---|---|
| entailed (21) | 21 | 0 |
| not-entailed (21) | 0 | 21 |
| partially-relevant (8) | 0 | 8 |

Against `design_intent` as a provisional oracle the proxy separates positives
from negatives cleanly; the partials all read as not-entailed (the proxy is
conservative on paraphrase without lexical overlap). **These are not accuracy
numbers** — accuracy and false-accept rate are computed against *human* labels,
which are pending.

## Metrics (computed once `human_label` is filled)

- **accuracy** = (verdict == human_label) / 50.
- **false-accept rate** = (verdict entail ∧ human_label not-entail) / (human_label
  not-entail) — the load-bearing number.

`tests/test_verify.py::test_verifier_accuracy_false_accept_rate` (currently
`xfail`) asserts a false-accept-rate bar once labels exist; flip it to a hard
gate in the Track C chat after labelling and after the `--neural` cache
regeneration.
