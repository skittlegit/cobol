## The novelty sentence (locked)

> **COBOL Archaeologist is the first benchmark and agentic system for
> temporal regulatory-drift detection: version-conditioned conformance
> judgments of legacy COBOL banking code against dated financial-regulation
> clauses, where the correct verdict for the same code changes with the
> regulation version — a task on which version-blind violation detectors
> fail by construction.**

Fallback claim (pre-authorized by the T4.5 decision rule): if the M4
oracle-slice comparison shows the loop adds nothing over the slicer, the
sentence reframes to "the first *benchmark* for temporal regulatory drift in
legacy COBOL, with interprocedural program slicing as the key enabling
capability" — benchmark-first, system-second. The benchmark half of the claim
survives either outcome.

What the sentence deliberately does NOT claim: not the first
compliance-detection-in-code benchmark (GDPR-Bench-Android is), not the first
financial-regulation × code dataset (Compliance-to-Code is), not the first
paired-version legal evaluation (temporal statutory QA is). The claim is the
intersection plus the temporal axis over *code*.

## Related work by cell

### Cell 1 — COBOL/mainframe × LLM (no regulation anywhere)

- **XMainframe / MainframeBench** (arXiv 2408.04660, FPT AI Center) — COBOL
  MCQ, QA, and summarization. Knowledge benchmark, no compliance task.
- **COBOLEval** (BloopAI 2024, github.com/BloopAI/COBOLEval) — 146
  HumanEval problems transpiled to COBOL; generation pass@1 via GnuCOBOL.
- **COBOLCodeBench** (Kumar 2025) — 46 BigCodeBench-Hard tasks in COBOL.
- **COBOL-Coder + COBOL-JavaTrans** (arXiv 2604.03986) — domain-adapted
  generation + bidirectional COBOL↔Java translation benchmark.
- **COBOLAssist** (arXiv 2604.03978) — compilation-error repair.
- **OpenCBS** (2022) and **X-COBOL** (2023) — defect suite / repo dataset.

Every task here is generation, translation, summarization, or generic
defects. **No work in this cell touches regulation.**

### Cell 2 — Regulation × source code (no COBOL, no temporal axis)

- **GDPR-Bench-Android** (arXiv 2511.00619) — **the nearest structural
  neighbor.** 1,951 annotated GDPR violations, 15 Android repos, 23 articles;
  localization via Accuracy@k at file/module/line + snippet multi-label
  macro-F1; 11 methods including a ReAct agent, RAG, and a formal AST
  baseline. Deltas that keep us novel: single fixed regulation version
  (violations, not drift); modern Android/Java-Kotlin, not legacy
  interprocedural COBOL; real-repo annotation, no synthetic mutation engine,
  no anti-gaming machinery, no verification tiers, no abstention. Notably
  their agent did NOT dominate (best at file level, beaten by a plain LLM at
  line level) — direct evidence our headline agent-vs-RAG question is open,
  not assumed.
- **CompliBench-Android** (arXiv 2511.00624, same group) — LLM compliance
  detection in Android; same deltas apply. (Name collision: a *different*
  CompliBench, arXiv 2604.12312, is dialogue-agent judging — unrelated.)
- **Compliance-to-Code** (arXiv 2505.19804) — Chinese financial regulations
  structured for compliance-checking **code generation**. Owns "financial
  reg × code" but generates checkers; does not detect drift in existing
  legacy code, and has no versioning.
- **LogiSafetyBench** (arXiv 2601.08196) — regulations → LTL oracles for
  agent *tool traces*, not source-code analysis.

### Cell 3 — Compliance violation detection over non-code artifacts

FinGuard-Bench (arXiv 2605.29427, LLM interactions), FinSafetyBench (arXiv
2605.00706, red-teaming refusals), FinRule-Bench (arXiv 2603.11339, financial
tables vs accounting rules), CompliBench-dialogue (arXiv 2604.12312). Useful
for framing "compliance detection is hard for frontier models"; none analyze
programs.

### Cell 4 — Temporal / versioned legal reasoning (text only)

- **Temporal statutory QA** (arXiv 2605.23497, TUM) — paired pre-/post-
  amendment questions diagnosing version-dependent answers. **This is the
  T6 design pattern in QA space** — T6 must cite it and position as the
  first version-paired evaluation over *code conformance* rather than text
  answers.
- **IndiaFinBench** (arXiv 2604.19298) — Indian financial regulatory text QA
  with amendment chains; finds temporal reasoning is the dominant error mode
  for top-tier models. Doubly useful: motivates the temporal axis AND is
  RBI-adjacent, so "RBI + temporal over text" is taken — our delta is
  binding it to code.
- **CALRK-Bench** (arXiv 2603.26332) — Korean law temporal context
  recognition. Same pattern, same cell.

### Cell 5 — Industrial incumbents (explain/migrate, not drift)

IBM watsonx Code Assistant for Z, **AWS Transform** (business-rule extraction
+ COBOL→Java over CardDemo itself), Google Mainframe Assessment Tool MCP,
and the community **Mainframe Architect MCP** (demoed on CardDemo — structure,
lineage, dead code). All confirm the playbook's premise: explanation and
migration are crowded and well-funded; **none bind code to regulation clauses
or versions.** This cell is why the contribution is detection, not migration.

## Gap statement

No existing benchmark or system evaluates whether an automated method can
judge legacy COBOL code against **versioned, dated** financial-regulation
clauses and detect **drift** — a conformance verdict that is a function of
`(clause version, effective_date)`. The nearest neighbor
(GDPR-Bench-Android) is static single-version violation detection on a
modern language; the temporal-pairing works are text QA; the COBOL
benchmarks are regulation-free. The intersection cell is empty as of
2026-07-07.

## Threats to the claim (reviewer-grade, be honest)

1. **Metric-design familiarity.** Accuracy@k localization + macro-F1
   classification + agent/RAG/formal baselines will read as
   GDPR-Bench-Android's design. Counter: cite it as the template we extend;
   our incremental metrics are T4-per-tier, T6, abstention handling, and the
   anti-gaming probe — lean on those in the paper, not on the shared skeleton.
2. **Active competitor group.** The GDPR-Bench/CompliBench-Android authors
   shipped two benchmarks in Nov 2025 and plausibly extend to other
   regulations or languages. The cell could close; schedule risk is real.
3. **"Temporal" must stay measured.** IndiaFinBench and the TUM work make
   temporal-legal claims common; ours is only defensible because T6 exists.
   If T6 gets cut, the novelty sentence collapses to Cell-2-with-COBOL —
   materially weaker. T6 is load-bearing; never trade it for scale.
4. **Agent superiority is empirically open.** GDPR-Bench-Android's agent
   lost at line level. Our headline bar (+10 F1 interprocedural, CI-backed)
   may fail; the T4.5 fallback framing exists for exactly this.
